## 1. MVP 재정의

현재 MVP는 다음 한 문장으로 고정한다.

> chord progression, BPM, density, energy를 입력받아 Stage A Music Transformer + LoRA 기반으로 1~2 bar symbolic MIDI solo phrase를 생성하고, repair/metrics/fallback을 통해 valid MIDI와 품질 지표를 반환하는 모델 inference pipeline.

이번 MVP에서 중요한 것은 실시간 DJ plugin이나 Spring Boot 백엔드가 아니다. 핵심은 모델 입력, 생성, 보정, 품질 gate, fallback, metrics가 하나의 반복 가능한 Python pipeline으로 동작하는 것이다.

## 2. What Works

- `scripts/generate.py`가 Stage A LoRA checkpoint와 conditioning MIDI로 샘플을 만든다.
- `inference/app/conditioning.py`가 request chord progression을 low-register chord guide MIDI로 변환한다.
- `inference/app/generator.py`가 model-first generation contract를 제공한다.
- 모델 후보 여러 개를 만들고, repair 후 metrics gate를 통과한 후보 중 quality score가 가장 낮은 MIDI를 선택한다.
- model output이 gate를 통과하지 못하면 fallback generator가 valid MIDI를 만든다.
- `temperature`, `top_k`, `top_p`, `model_candidates`가 request에서 model generation까지 연결되어 있다.
- `inference/app/model_runner.py`가 모델을 한 번 로드하고 여러 request에서 재사용할 수 있다.
- FastAPI wrapper는 최소 `/health`, `/metadata`, `/infer/midi`를 제공한다.
- 기본 MVP inference는 `max_sequence=256`으로 낮춰 512-token 생성보다 빠른 기준선을 갖는다.
- 단위 테스트가 request conditioning, parameter validation, candidate scoring, in-process runner path를 확인한다.

## 4. What Is Fragile

- request-derived conditioning은 실제 control-token 학습이 아니라 MIDI primer hack에 가깝다.
- postprocess가 pitch range, phrase alignment, gap fill을 많이 담당한다.
- dense/medium은 repair 덕분에 안정적이지만, 이 안정성이 모델 자체의 순수 생성 능력이라고 말하면 안 된다.
- sparse는 긴 공백을 허용하도록 gate를 조정했지만, 실제로 음악적으로 좋은 sparse phrase인지 수동 청취/DAW 확인이 필요하다.
- 현재 metrics는 유용한 proxy지만 음악적 완성도를 충분히 평가하지 않는다.
- `decode_midi`에서 많은 `info removed pitch` 로그가 나온다. token distribution이 여전히 noisy하다는 신호다.
- FastAPI는 thin wrapper 수준이다. async job, queue, DB는 이번 모델 MVP 범위가 아니다.
- 9~10초/request는 CLI/API demo에는 가능하지만 live co-performer에는 부족하다.

## 5. What Is Not Real ML Yet

현재 시스템은 ML model serving pipeline에 가깝지만, 아직 강한 의미의 chord-conditioned improvisation model은 아니다.

- chord progression은 학습된 symbolic chord token이 아니라 generated conditioning MIDI로 들어간다.
- section, energy, density는 모델 내부 control token이 아니라 sampling, repair, fallback에서 주로 반영된다.
- style personalization은 LoRA checkpoint에 일부 기대고 있지만, 명확한 style adapter evaluation은 없다.
- phrase quality는 모델 단독 출력보다 repair/fallback contract로 보장된다.

따라서 외부 설명에서는 다음처럼 말하는 것이 정확하다.

> Stage A is a model-first symbolic MIDI generation pipeline with request-derived MIDI conditioning, post-processing, quality gating, and fallback generation.

아직 다음처럼 말하면 과장이다.

> Fully chord-aware personalized jazz improvisation model.

## 6. What To Keep

- 모델 MVP 범위는 Python CLI/FastAPI까지만 유지한다.
- Spring Boot, DB, async job server는 모델 MVP 이후 portfolio extension으로 둔다.
- `python -m inference.app.generator`를 primary contract로 둔다.
- `scripts/generate.py`는 Stage A raw generation utility로 유지한다.
- `music_transformer/generate.py`는 legacy entrypoint로 취급한다.
- metrics JSON과 sweep markdown을 계속 남긴다.
- 모든 큰 변경은 sweep 결과와 함께 문서에 기록한다.

## 7. Decision Log

- MVP primary interface: Python generation contract.
- FastAPI: optional thin wrapper.
- Inference default: `max_sequence=256`.
- Stage A comparison mode: `max_sequence=512`.
- Sampling baseline: `temperature=0.9`, `top_p=0.95`, `model_candidates=2`.
- Current success target: valid MIDI + model success rate + fallback explainability.
- Realtime latency target은 현재 phase의 acceptance gate가 아니다.
